### merge commit vs squash merge vs rebase merge

## merge commit

- I learned that in merge commit _all the commit history is preserved_.

- The changes from the branches are merged into the main branch and a _new merge commit_ that records the point where two histories were joined .

- And the merge commit will have two parents one from the main and the another from the branch .

- They problem with this merge is that because of keeping all the history many branches makes the history messy .

## Rebase merge 

- Rebase takes the changes from the branch and replays them on the top of the latest main which makes it look like that the work started from the newest version main.

- This method of committing solves the problem of making the history messy cause this one makes it linear.

- But the problem with this method is commit is that it changes the history
  - old commits replaced by new commits new commit created by Git.

## squash merge

- This works like the name says it squash everything to one commit

- Eg if we made small 5 commits and if i don't want them all in there we can use squash

- It makes the individual commits to disappear [combines them to into single new commit ]

  In conclusion I learned that :

_Merge → creates one new merge node._

_Rebase → creates new replacement nodes for each rebased commit._

_Squash → creates one new node that replaces multiple commits_
